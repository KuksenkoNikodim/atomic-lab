# Анализ рисков подхода с неатомарными коммитами

## Риски:

1. **Сложность отката (rollback)**
   - При необходимости откатить одно изменение придется откатывать все изменения в коммите
   - Невозможно выборочно отменить конкретную функциональность

2. **Проблемы при код-ревью**
   - Сложно оценить каждое изменение по отдельности
   - Reviewer вынужден анализировать разнородные изменения в одном коммите
   - Увеличивается время на ревью и вероятность пропуска ошибок

3. **Риск «снежного кома» зависимостей**
   - Изменения в разных частях кода могут создавать скрытые зависимости
   - При merge конфликты сложнее разрешать из-за смешанных изменений
   - Трудно отследить, какое изменение привело к багу

4. **Проблемы с git bisect**
   - При поиске бага через бинарный поиск сложнее определить виновный коммит
   - Коммит содержит много изменений, что затрудняет локализацию проблемы

5. **Отсутствие четкой истории**
   - Невозможно понять контекст каждого изменения
   - Затруднена документация через историю коммитов
